Appendix to Part A

Onboarding a client: intake to configuration

Part A, question 2 claims a fourth client is authored, not coded. That claim only holds if there is a disciplined way to turn "how this client works" into configuration — and a clear point where a requirement stops fitting and must escalate instead of becoming a fork. This is that process. It reuses the Part B definition format throughout, and the same validateRecord from Part B is the tool I use to check I understood the client before anyone builds.

Understand→ Validate→ Consolidate→ Configure→ Escalate (only if it doesn't fit)

This appendix is deliberately short. It exists to make one thing concrete — the operational side of "no per-client code" — not to lengthen the submission.

1Intake — the questions, and where each answer lives

Every question below is chosen because its answer has a home in the config model from Part A, question 3. If a client tells me something that has nowhere to go, that is the signal for section 3 — not a reason to open the code.

2Validate before building — the step people skip

I do not trust a definition I have drafted until it survives the client's own history. The intake produces a draft record_definition. Before a line of product code is written, I take a sample of the client's real past records and run them through validateRecord against that draft — the exact library in Part B.

Records the client considers good must come back valid: true. Records their staff would have rejected must fail, and the message has to read correctly to those staff. Every mismatch means one of two things is wrong — my understanding, or the definition — and it is caught in a spreadsheet review with the client, not in production three weeks later.

This is why Part B returns structured, human-readable issues rather than a boolean: the validation output is the artefact I walk the client through to confirm we agree on what their data means. Consolidation is not a meeting; it is a passing test suite the client has signed off on.

3The escalation boundary — where intake stops and honesty starts

Some answers have nowhere to land. Intake's real job is to find these early and name them, rather than quietly bending the code to fit:

  • A rule between two fields ("project end must be after project start") — the format cannot express it today. Small declarative cross-field rules, added to the format for everyone. (Part B README.)
  • A computed value — Client B's eligibility score calls an external API and runs a board-revised formula. That is a program, not configuration. It goes to the one named, versioned computation extension point, with its parameters kept in config. (Part A, question 8.)
  • A rule that depends on the answer ("guardian required if patient is a minor") — the same class of gap; extend the vocabulary or decline, never special-case.

The discipline: the boundary is crossed in the open, in a named place, for every client at once — never with an if (tenant === …).

The intake above is, in effect, a form for populating these tables. This is the concrete shape behind Part A questions 3, 4 and 6 — relational skeleton, client-shaped record payloads in JSONB, every row scoped by tenant_id under row-level security.

Why records are documents (JSONB) inside a relational skeleton, and why the whole thing sits behind row-level security rather than a database per tenant, is argued in Part A, questions 4 and 6. This appendix shows only how a client conversation fills it.

The schema, as DDL

The same tables as runnable Postgres — the versioned JSONB definition store, the record store it stamps, and the row-level-security policy that is the isolation guarantee. Full file: architecture/schema.sql.

the load-bearing tables + the RLS policy 
-- Field definitions: the Part B format, versioned, immutable once published.
create table record_definitions (
  id           uuid primary key default gen_random_uuid(),
  tenant_id    uuid not null references tenants(id),
  record_type  text not null,
  version      int  not null,
  status       text not null default 'draft'
                 check (status in ('draft','published','retired')),
  fields       jsonb not null,        -- the fields[] array validateRecord reads
  unique (tenant_id, record_type, version)
);

-- Submitted records: documents, tagged with the definition version that
-- validated them, so old records keep their original meaning.
create table records (
  id             uuid primary key default gen_random_uuid(),
  tenant_id      uuid not null references tenants(id),
  definition_id  uuid not null references record_definitions(id),
  record_type    text not null,
  status         text not null,
  data           jsonb not null
);

-- Row-Level Security: the tenancy guarantee. Applied to every tenant-scoped
-- table; a query that forgets a tenant filter returns zero rows, not a breach.
alter table records enable row level security;
create policy records_tenant_isolation on records
  using      (tenant_id = current_setting('app.tenant_id')::uuid)
  with check (tenant_id = current_setting('app.tenant_id')::uuid);
