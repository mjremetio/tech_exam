Part AArchitecture note

1Questions I would ask before building

Who makes configuration changes once we're live — our staff, or the clients themselves? The whole brief turns on "constant small changes"; whether a renamed status is a support ticket to us or a self-service edit by the client changes what the admin tooling must be.
Assumption: our staff, through an internal admin interface, for at least the first year. Client self-service is a later product decision, not a foundation requirement, but the config model must not preclude it.

What happens to the three existing systems and their data? Migrating years of resident history and grant records is a project of its own, and Client A's "show me this resident's history" only works if the history comes across.
Assumption: we migrate active records and enough history to serve the features that need it; the old systems stay readable but frozen until decommissioned.

Where must the data physically live, and who is the legal data controller? Client C is under Israeli privacy law and expects an audit; that may constrain hosting region, retention, and deletion obligations for everyone sharing the platform.
Assumption: each client is the controller and we are the processor; a single hosting region acceptable to all three (an Israeli or EU region) is workable, and I would confirm the specifics with a lawyer rather than guess. The design keeps per-client data export and deletion cheap either way.

What are the availability and urgency expectations, concretely? "Duty nurse is alerted within four hours" implies someone is watching alerts at night. Is this platform life-safety-adjacent, or business-hours software with one time-sensitive rule?
Assumption: business-hours criticality with one hard timing rule (the four-hour triage alert), which gets a monitored, redundant delivery path — not a platform-wide 24/7 SLA.

How exact does Client A's resident matching need to be? Same person, different phone numbers, misspelled names: that is identity resolution, and fully automatic merging of people's records is both hard and risky.
Assumption: the system suggests likely matches and staff confirm them. It never merges records on its own.

Must past eligibility scores be reproducible after the board changes the formula? If a rejected applicant challenges a decision a year later, can we show the score, the formula version, and the inputs that produced it?
Assumption: yes. Formulas and per-round weights are versioned, and every stored score records which version produced it from which inputs.

What is the billing model? Flat per-client subscription versus usage-based pricing changes how much per-tenant metering the foundation needs from day one.
Assumption: flat subscription; no metering in the foundation.

2How would you model "a client's data shape"?

A client's data shape is data, not code. Each client has one or more record types (maintenance report, grant application, patient referral), and each record type is described by a definition: an ordered list of fields with a name, label, type from a fixed vocabulary (text, number, date, choice, file…), a required flag, options, constraints, and a sensitivity level. This is exactly the format in the starter package, and Part B is the working proof: one validator and one form-describer, driven entirely by definitions, handling three very different clients with zero client-specific code.

Submitted records are stored as documents (JSONB in Postgres) alongside the id of the definition version they were validated against. Definitions are versioned and immutable once published, so old records keep meaning what they meant when they were submitted; that matters to a clinic facing an audit.

For onboarding client four to require no new code, three things have to be true:

  • Nothing downstream names a field. Validation, form rendering, list screens, filters, exports and reports all read the definition at runtime. If any screen hard-codes "neighbourhood", the game is lost.
  • The field-type vocabulary is closed and generic. New clients combine existing types; they do not get new ones. Adding a type is a platform change that benefits everyone at once.
  • Everything else a client varies — statuses, roles, notifications, queues — is configuration in the same spirit (question 3), so "onboard a client" means "author configuration", a task for a person with an admin screen, not a programmer with a branch.

3Where do client differences live?

In a set of versioned configuration tables in the platform database, scoped by tenant, read at runtime, and cached. Concretely, per tenant:

  • record_definitions — the field definitions above, as versioned JSONB rows with draft → published states.
  • workflows — the status list and the allowed transitions, each transition naming which roles may perform it. Client A's "only a department head may assign" and Client C's "only a triage nurse may set urgency" are rows here, not code.
  • roles — named per tenant, mapping to permissions over record types, transitions, and fields (reception's inability to read clinical notes is a field-level read permission).
  • notification_rules — "on entering status X, send template Y over channel Z to recipient R, with optional delay". Client B's "second email fourteen days later" is a rule with delay: 14 days; the four-hour triage alert is the same shape with a timeout trigger.
  • views — the queue screens: which filters, which sort. "Urgency then arrival time, filtered by specialty" is a stored view description the frontend renders generically.
  • templates — email/SMS wording with placeholders resolved from record fields.

The mechanism: services read config through one config module that caches per (tenant, config-kind) and invalidates on publish. Publishing a config change is an audited event. Under constant small change requests, the configuration is the product, and "who changed the email wording and when" must be answerable.

4Keeping clients apart

Single database, shared schema, tenant_id on every row — enforced by Postgres Row-Level Security. Every connection sets the tenant context from the authenticated session before any query runs, and RLS policies make other tenants' rows invisible to every query. The guarantee lives in the database, beneath the application: a developer who forgets a WHERE tenant_id = clause gets an empty result, not a breach.

Layers above repeat the guarantee rather than relying on it:

  • The tenant id comes only from the server-side session established at login — never from a request parameter, header, or URL a client could influence.
  • The application's data-access layer scopes every query by tenant anyway; RLS is the net under it, not the only wall.
  • Files live in per-tenant prefixes in object storage, reachable only through short-lived signed URLs minted by the backend after the same permission checks.
  • An automated test suite logs in as tenant A and attempts every read against tenant C's known data, and fails the build on anything but empty results.

Field-level sensitivity is part of the same enforcement point: the API strips fields the caller's role may not read (clinical notes, for reception) at the server, so the data never reaches a browser that shouldn't hold it. Hiding fields client-side would only be cosmetic.

5What breaks first at 300 clients?

The queue screens' queries against the shared JSONB record store. Every client's staff works all day from a filtered, sorted list (the single hottest path in the product), and the filters are client-defined at runtime, so they hit JSONB paths the database has no indexes for. At three tenants a sequential scan over a few hundred thousand rows is invisible. At 300 tenants, with a few Client-A-sized tenants at 200+ records a day, storm bursts of 4,000, and years of accumulation, those scans are the first thing users feel, and because the store is shared, one heavy tenant degrades everyone's queues at once. It fails first because it combines the highest query frequency, the least statically-known shape, and the shared blast radius.

What I would do, in order: when a definition is published, generate indexes from the config: the definition already declares which fields are filterable and sortable, so create Postgres expression indexes (or generated columns) on those JSONB paths per tenant's record type, and partial indexes on open statuses since queues only show open items. That is the config-driven philosophy applied to the physical layer, and it is cheap. After that: read replicas for queue reads, and, once a tenant outgrows shared hardware, moving that tenant's partition to its own node, which the tenant-scoped model allows without redesign.

Runner-up: at 300 clients × constant small changes, the humans processing config requests break before the hardware does. The mitigation is the same config model exposed as client self-service — which is why question 1's first assumption matters.

6Technology choices

Languages. TypeScript everywhere: backend, frontend, and the shared definition/validation library. That follows directly from the architecture: validation driven by definitions should run identically in the browser (instant field feedback) and on the server (the actual gate), and one language means it is literally the same code, not two implementations drifting apart. SQL is the second language, written deliberately for reports and the queue paths.

Backend & frontend. Node.js with Fastify, a thin and fast HTTP layer; this system's complexity belongs in the domain model, not the framework. Frontend: React + Vite, rendering forms and queue screens generically from the Part B form descriptions and the stored view configs. No per-client frontend code.

Database. PostgreSQL, one database doing double duty: relational tables for the parts that are genuinely relational (tenants, users, roles, workflows, transitions, audit log, notification rules) and JSONB for the client-shaped record payloads. This problem is the case JSONB-in-Postgres exists for: per-client variable shapes inside a fixed relational skeleton, with row-level security, real transactions (a status change + audit row + outbound notification enqueue must commit atomically), and SQL aggregation for every client's reporting needs.

Third-party services. Buy the undifferentiated parts:

Hard to reverse: the definition format itself (every stored record embeds its assumptions; this is the choice to get right), Postgres-with-RLS as the tenancy model, and TypeScript-everywhere. Easy to swap: Postmark, Twilio, Sentry, Grafana (all behind thin adapters; email and SMS senders are interfaces from day one), hosting (nothing AWS-specific above the infrastructure layer), even Fastify at moderate cost. Auth0 sits in the middle: swappable in principle, tedious in practice, which is why sessions and roles stay in our model with Auth0 only proving identity.

The alternative I took seriously: MongoDB for the record store — documents with per-client shapes are its home ground, and half this design looks like a document problem. I rejected it because the decisive requirements are the ones around the documents: row-level security as a database-enforced tenancy guarantee, atomic transactions across records + audit + workflow state, and ad-hoc SQL aggregation for every client's reports. Postgres JSONB gives me roughly ninety percent of Mongo's flexibility while keeping all three; Mongo would give me ideal document ergonomics and make me rebuild isolation, transactional discipline, and reporting myself. I also simply know Postgres's failure modes better, and per the brief that counts, so I'm counting it.

7Five principles for the development itself

1. Client differences are configuration; if satisfying a change request requires a deploy, the platform has a bug. This is the direct answer to "constant small change requests". It settles arguments: when a developer proposes handling a client ask with a quick conditional, the principle says the correct fix is config, or extending what config can express.

2. No code may name a client. No if (tenant === "clinic"), ever, enforced in review and greppable in CI. The moment one branch exists, the second is easier, and the platform starts becoming three systems in one repository again, which is the disease we're escaping. Part B is this principle in miniature.

3. When config can't express a client's need, extend the vocabulary for everyone or decline — never special-case. The hardest moments will be requests that almost fit. This principle forces the honest question "is this feature general?" and gives a legitimate answer of "no, and therefore we won't" — which principle 1 alone doesn't.

4. Authorisation happens server-side, at field granularity, or it hasn't happened. Reception must not see clinical notes; a role that "can do most of what another can but not all" is the brief's own example. UI hiding is not enforcement. Any disagreement about where a permission check belongs is settled: at the API, before data leaves the server.

5. Every mutation — of records, and of configuration — is an append-only audit event with an actor. The clinic will be audited; the board revises formulas; statuses are renamed weekly. In a platform where behaviour is data, "who changed what, when" about config is as important as about records. It also settles the debugging argument: history is never reconstructed from guesswork, it is read.

8What didn't fit?

Client B's eligibility scoring. Statuses, fields, notifications and queues all reduce to configuration a non-programmer could author. The score does not: it calls an external registry API, computes a ratio, applies recency-weighted grant history, applies board-set thematic weights, auto-rejects below a threshold — and the board rewrites the formula itself yearly. That is a program, and pretending a config format can express it leads to a homemade expression language that is worse than code: untyped, untested, and impossible to debug.

What I would do: admit it is code, and quarantine it. The platform gets a computation extension point: a versioned module with a narrow contract (record in, score + explanation out), written and deployed by us, with its parameters — weights, thresholds, per-round settings — kept in config where the board's annual changes usually land. Every score stores the module version and inputs that produced it, for the audit-trail reasons in question 1. This is the one place the "no client-specific code" rule bends, and the design is honest about it: bent in one narrow, named, versioned place instead of silently everywhere.

Half-fits, for completeness: Client A's resident identity resolution is genuinely hard but at least generic (a platform "possible duplicate" feature, suggest-and-confirm); cross-field validation rules ("project end after project start") don't fit today's definition format and I'd extend it with a small declarative rule set — discussed in the Part B README.

9What would you build first?

First: the definition-driven record core, carrying one real client end-to-end. Concretely: tenancy with RLS and the auth integration; the definition store with versioning; the validation library (Part B, essentially) enforcing definitions at the API; record storage; a generic form rendered from the form description; a generic queue screen driven by a view config; and the append-only audit log. Then onboard Client A as configuration only and let real 200-a-day traffic hit it; it has the simplest workflow and the highest volume, with none of the medical-grade sensitivity to get right in week one.

Two things are in the core that people often defer: isolation and audit. Neither can be retrofitted honestly: bolting RLS onto a live schema, or reconstructing an audit trail after the fact, is the kind of quiet papering-over this platform exists to end.

Later, by choice: the notification rules engine (Client A's single Resolved-SMS gets a minimal hard-shaped version first; the general rule engine comes with Client B's delayed emails); eligibility scoring (needs the extension point, and rounds are quarterly, so there is calendar time); reporting dashboards (the director's totals start as plain SQL behind an endpoint); resident identity matching (suggest-only, after real duplicate data exists to tune against); client self-service admin (our staff author config through internal screens first); and Client C last among the three, not because it's least important but because its field-level permissions and audit expectations deserve a platform that has already carried live traffic.

Part BCoding exercise

Code: github.com/mjremetio/tech_exam

A TypeScript library, roughly 600 lines plus tests: validateRecord(definition, record) returns structured errors and warnings; describeForm(definition) returns a renderer-agnostic form description. No client knowledge anywhere in the code: all three sample clients (and a fourth invented in the tests) go through the same paths. 30 tests cover every sample record and the awkward cases the samples push on: false and 0 as real values, empty string vs null vs absent, stringly-typed numbers and booleans, impossible calendar dates, unknown extra fields, and broken definitions.

The README in the repository covers the decisions the format left open (no type coercion; unknown fields warn rather than block; empty-anything is "missing"), what the format cannot express (cross-field rules like "project end after start", found via sample record B-2), and the full example of the Part 2 output.

Part CClosing reflection

What I left out on purpose. There is no workflow engine design in detail, no API surface, no UI sketches, and the notification engine is described in one sentence. The exercise asks for a foundation, and I spent the space on the four decisions that are expensive to reverse: the definition format, the tenancy model, the config-not-code boundary, and where that boundary breaks (scoring). Everything I left out can be built on top of those without contradiction; nothing I left out could repair them if they were wrong.

What I noticed and chose not to address. The storm-day burst of 4,000 reports is less a database problem than an SMS-cost and queue-backpressure problem; it is why sending is a background job rather than an inline call, but I did not size it. Client C's "subject to Israeli privacy law" almost certainly implies specific obligations (retention, breach notification, possibly residency) that I flagged as a question rather than pretending to know; designing to a law I haven't read would be exactly the quiet papering-over the brief warns against.

The decision I am least confident about: shared-schema multi-tenancy with RLS, rather than schema-per-tenant or database-per-tenant. I believe it is right for operational sanity at hundreds of small tenants (one migration, one backup story, cross-tenant ops stay possible), and RLS makes the isolation real. But a clinic audit, a regulator, or a large client demanding physically separate storage would change my mind, and I would want that conversation before 300 tenants share hardware, not after. The design hedges: tenant-scoped everything means extracting one tenant to its own node is migration work, not redesign.

What I would do differently starting again. In Part B, I would design the error model around a message catalogue from the first line instead of baking English strings in and exposing codes alongside; the codes are there, but the seam shows. In Part A, I spent early effort on the scoring problem before realising the honest answer was "this doesn't fit, and here is the quarantine" — I would reach that faster by asking "could an admin screen express this?" of every requirement at the start.

What I wanted to build and didn't get to: the declarative cross-field constraint extension for the definition format, with validation of the definitions themselves. That is the format gap sample record B-2 (a project that ends before it starts) is pointing at.

AIHow I used AI on this exercise

Tools and models. I used Claude (Anthropic), through the Claude desktop app, as the primary tool across the whole exercise, and Claude Code in my editor at the end, to run the tests and demo on my own machine before submitting. No other models or AI tools were involved.

What I used it for. Reading the brief and starter package closely (including having it enumerate what each sample record was designed to test); thinking through the architecture questions in long back-and-forth before writing the answers; drafting and iterating the Part B library and its tests; drafting the prose of this page; and building the page itself.

How I worked with it. Mostly extended back-and-forth rather than single prompts: I gave it the full brief and starter data up front, made the framing decisions (language, the config-not-code stance, no type coercion), and had it draft, then challenged the drafts — asking it to argue against its own choices, tighten answers that were padding, and justify each validation decision against a specific sample record. Nothing this page claims about the code went out unverified: before submitting I ran the full test suite and the demo over all eleven sample records on my own machine, and checked that the README's example output matches what the code actually prints.

Something I rejected, and why. Claude's first instinct on unknown extra fields was strict: any key the definition didn't name should make the record invalid. I rejected that. Record A-2's note says some of its contents "look like problems but are not", and operator_initials reads like a leftover from the old per-client system, the kind of thing a migration or an integration will send. Blocking the record punishes the submitter for the platform's own stale definition; silently dropping the field hides a real signal that the definition is out of date. So unknown fields became warnings: the record goes through, the drift is surfaced. If a client needed strictness (a clinic under audit, say) I would make it a per-definition setting rather than a global rule.

Where it helped least. Getting the code running on my actual machine. The model reasons well about code it can see, but it cannot see a laptop: my default Node was too old for the toolchain, and a dependency install built for the wrong CPU architecture had to be thrown away and redone before the tests would start. Every instruction sounded authoritative, and each failed against some local fact the model had no way to know, so that stretch was ordinary debugging with a confident narrator. The pattern across the exercise: it helped most where its claims could be checked by running something, and least where the ground truth lived outside the conversation.
